Why this exists

"Memorized diagrams collapse the moment a constraint changes."

The problem with memorizing architectures

A diagram you memorized has no reasoning underneath it

  • Finished diagrams teach the answer, not the forces that produced it.
  • Change one constraint — and the picture breaks with nothing to rebuild from.
  • The repo's GUIDE names 10 failure modes that follow from this trap:

↳ Rushing to design before clarifying requirements

↳ Weak trade-off articulation · no sense of numbers

↳ Ignoring failure modes & degradation

↳ Over-indexing on the "correct" architecture

↳ Treating blocks as opaque primitives

↳ Can't course-correct when constraints change

The idea

Building blocks you compose

One skill per part of a system — a divide-and-conquer wiki. Each block is a recipe carrying four things:

Explicit trade-offs

Every option labeled with what it solves, what it worsens, and when to change it.

Behavior under stress

How the component degrades and recovers when load, regions, or nodes fail.

The numbers that matter

QPS, storage, latency, fan-out — the figures that force the architecture.

Provider variants

Generic-first recipe, then AWS / Azure / GCP / Temporal managed mappings.

How it works

A GUIDE-driven reasoning loop — iterative, not a script.

The reasoning loop

Reason, don't memorize — and iterate

A constraint changed? Re-enter at the affected phase. Iterate → Clarify never stops.

The six phases

One line each

  • 1 · Clarify — functional & non-functional requirements, and scope.
  • 2 · Estimate — peak QPS, storage, bandwidth, server count. Numbers force the architecture.
  • 3 · Design — compose the building blocks the numbers demand.
  • 4 · Trade-offs — every choice as solves / worsens / when-to-change.
  • 5 · Failure modes — SPOFs, how it degrades, how it recovers.
  • 6 · Iterate — name the invalidated assumption, redesign only the affected part.

Design principles & guardrails

The rules underneath

  • Clarify before designing · quantify before choosing.
  • Justify every choice — solves / worsens / when-to-change.
  • Design for failure — stale beats error.
  • Treat every architecture as a hypothesis that holds until a constraint changes.
  • Provider files are generic-first — never an invented service.
